# Feature Specification: SpeckitUI Core Application

**Feature Branch**: `001-speckitui-core`  
**Created**: January 16, 2026  
**Status**: Draft  
**Input**: User description: "SpeckitUI - A desktop/web UI application for the spec-kit spec-driven development workflow. Features a navigation pane with workflow steps (Specify, Plan, Tasks, Implement, Test, Push, PR, Bug Fix), detail panes showing artifacts and checklists for each step, project/repo container with multiple spec instances, terminal window area for CLI tools, and GitHub integration for PR feedback and issue tracking."

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Navigate Workflow Steps (Priority: P1)

As a developer, I want to see and navigate between the different steps of the spec-driven development workflow so that I can understand where I am in the process and move between stages efficiently.

**Why this priority**: Navigation is the foundational UI element that enables all other interactions. Without workflow step navigation, users cannot access any feature functionality.

**Independent Test**: Can be fully tested by launching the application and clicking through each workflow step in the navigation pane. Delivers core navigation value immediately.

**Acceptance Scenarios**:

1. **Given** the application is launched, **When** the user views the navigation pane, **Then** they see workflow steps: Specify, Plan, Tasks, Implement, Test, Push, PR, Bug Fix displayed as selectable items
2. **Given** the navigation pane is visible, **When** the user clicks on a workflow step, **Then** that step becomes highlighted/selected and the detail pane updates to show that step's content
3. **Given** a workflow step is selected, **When** the user clicks a different step, **Then** the selection moves to the new step and the detail pane content changes accordingly
4. **Given** the user is on any workflow step, **When** they look at the navigation, **Then** they can see visual indicators showing which steps have content/artifacts and which are empty

---

### User Story 2 - View Spec Instance Artifacts (Priority: P1)

As a developer, I want to view the artifacts generated by each workflow step for my current spec instance so that I can review requirements, plans, tasks, and other specification documents.

**Why this priority**: Viewing artifacts is the core value proposition - users need to see their work to do anything meaningful in the application.

**Independent Test**: Can be tested by opening a project with existing spec artifacts and verifying each step displays the correct markdown files and content.

**Acceptance Scenarios**:

1. **Given** a spec instance exists with a spec.md file, **When** the user selects the Specify step, **Then** the detail pane displays the spec.md content in a readable format
2. **Given** a spec instance has plan.md, research.md, and data-model.md, **When** the user selects the Plan step, **Then** all related artifacts are accessible in the detail pane
3. **Given** a spec instance has tasks.md, **When** the user selects the Tasks step, **Then** the task list is displayed with checkboxes visible and their current state shown
4. **Given** an artifact contains markdown with headers, lists, and code blocks, **When** displayed in the detail pane, **Then** the markdown is rendered in a readable, formatted manner

---

### User Story 3 - Manage Project and Spec Instances (Priority: P1)

As a developer, I want to open a project/repository and select from multiple spec instances so that I can work on different features within the same project.

**Why this priority**: Multi-spec management is essential for real-world use where developers work on multiple features. This enables the container/instance model described in requirements.

**Independent Test**: Can be tested by opening a project with multiple spec directories and switching between them.

**Acceptance Scenarios**:

1. **Given** the application is launched, **When** the user opens a project folder, **Then** the application scans for spec instances in the `specs/` directory
2. **Given** a project has multiple spec instances (e.g., `001-auth-feature`, `002-dashboard`), **When** the user views the spec selector, **Then** all available spec instances are listed
3. **Given** multiple specs exist, **When** the user selects a spec instance, **Then** that spec becomes the active context and all workflow steps show artifacts for that spec
4. **Given** no spec instances exist in a project, **When** the user views the spec selector, **Then** they see an empty state with guidance on creating a new spec

---

### User Story 4 - Track Progress with Checklists (Priority: P2)

As a developer, I want to see and interact with checklists embedded in spec artifacts so that I can track my progress and validate quality at each step.

**Why this priority**: Progress tracking is a key differentiator that transforms static documents into actionable work items. High value but depends on artifact viewing (P1).

**Independent Test**: Can be tested by opening a spec with tasks.md containing checkboxes and verifying interactive toggling and progress display.

**Acceptance Scenarios**:

1. **Given** the Tasks step is selected and tasks.md contains checkboxes, **When** the detail pane renders, **Then** checkboxes are displayed as interactive elements
2. **Given** a checklist item is unchecked, **When** the user clicks the checkbox, **Then** the item becomes checked and the change is saved back to the markdown file
3. **Given** a step has multiple checklist sections, **When** viewing the step, **Then** a progress indicator shows completion percentage (e.g., "12/20 tasks complete")
4. **Given** plan.md contains Phase -1 Gates checklists, **When** the Plan step is selected, **Then** the gate checklists are prominently displayed with their completion status

---

### User Story 5 - Use Integrated Terminal (Priority: P2)

As a developer, I want to open and use terminal windows within the application so that I can run CLI commands including coding agent CLIs without leaving the environment.

**Why this priority**: Terminal integration enables the "inner + outer loop in one environment" vision. Important for workflow continuity but not required for basic artifact viewing.

**Independent Test**: Can be tested by launching a terminal, running a command, and verifying output is displayed correctly.

**Acceptance Scenarios**:

1. **Given** the terminal area is visible, **When** the user requests a new terminal, **Then** a new terminal session opens within the application
2. **Given** a terminal is open, **When** the user types a command and presses enter, **Then** the command executes and output is displayed in the terminal
3. **Given** multiple terminals are open, **When** the user views the terminal area, **Then** they can switch between terminals using tabs or a selector
4. **Given** a terminal is running a long process, **When** the user switches to another terminal, **Then** the first terminal continues running in the background

---

### User Story 6 - Review GitHub PR Feedback (Priority: P2)

As a developer, I want to see GitHub pull request feedback for my feature branch so that I can address review comments without leaving the application.

**Why this priority**: GitHub integration for PRs extends the workflow into the "outer loop" of development. Valuable for complete workflow but depends on core navigation and artifact viewing.

**Independent Test**: Can be tested by connecting to a GitHub repository with an open PR and verifying comments are displayed in the PR step.

**Acceptance Scenarios**:

1. **Given** the current spec branch has an open PR, **When** the user selects the PR step, **Then** the PR details and review comments are displayed
2. **Given** PR review comments exist, **When** viewing the PR step, **Then** comments are shown with their file context, author, and status (pending/resolved)
3. **Given** a PR has multiple review comments, **When** the user clicks on a comment, **Then** they can see the full comment context and related code
4. **Given** the PR has status checks, **When** viewing the PR step, **Then** the status of CI/checks is visible

---

### User Story 7 - Manage GitHub Issues for Bug Fixes (Priority: P3)

As a developer, I want to pull down and work on GitHub issues for bug fixes so that I can manage bug fix workflows within the same environment.

**Why this priority**: Bug fix workflow is a secondary use case that extends the SDD workflow. Lower priority as it's supplementary to the main specification-driven flow.

**Independent Test**: Can be tested by connecting to a repository with issues and verifying issue details are displayed in the Bug Fix step.

**Acceptance Scenarios**:

1. **Given** the Bug Fix step is selected, **When** the repository has open issues, **Then** available issues are listed with title, number, and labels
2. **Given** issues are displayed, **When** the user selects an issue, **Then** full issue details including description and comments are shown
3. **Given** an issue is selected, **When** the user chooses to work on it, **Then** they can create or switch to a branch for that issue

---

### User Story 8 - Access Constitution/Settings (Priority: P3)

As a developer, I want to access and view the project constitution and settings so that I can understand the project's governing principles and coding standards.

**Why this priority**: Constitution access is a reference feature used occasionally, not part of the main workflow loop.

**Independent Test**: Can be tested by opening a project with constitution.md and verifying it's accessible from a settings/preferences area.

**Acceptance Scenarios**:

1. **Given** a project has `.specify/memory/constitution.md`, **When** the user opens settings/preferences, **Then** the constitution content is viewable
2. **Given** the constitution has multiple articles, **When** viewing the constitution, **Then** articles are displayed in a navigable format
3. **Given** constitution_update_checklist.md exists, **When** viewing constitution settings, **Then** the update checklist is accessible

---

### Edge Cases

- What happens when a project has no `specs/` directory? → Display empty state with instructions
- What happens when a spec directory exists but is missing expected files (e.g., no spec.md)? → Show partial content with indicators for missing artifacts
- What happens when markdown files contain malformed syntax? → Render as best effort with fallback to raw text display
- How does the system handle very large markdown files? → Use virtualized rendering to maintain performance
- What happens when GitHub API authentication fails? → Show clear error state in PR/Issues steps with re-authentication option
- What happens when the terminal process crashes? → Allow user to close and open a new terminal
- How does the system handle concurrent edits to the same file? → Save triggers a file sync; warn if external changes detected

## Requirements *(mandatory)*

### Functional Requirements

**Navigation & Layout**
- **FR-001**: System MUST display a navigation pane with workflow steps: Specify, Plan, Tasks, Implement, Test, Push, PR, Bug Fix
- **FR-002**: System MUST highlight the currently selected workflow step in the navigation pane
- **FR-003**: System MUST display a detail pane that shows content relevant to the selected workflow step
- **FR-004**: System MUST include a terminal area that can display one or more terminal sessions
- **FR-005**: System MUST provide a way to access Constitution/Settings separate from workflow steps

**Project & Spec Management**
- **FR-006**: System MUST allow users to open a project folder/repository as the working context
- **FR-007**: System MUST scan opened projects for spec instances in the `specs/` directory
- **FR-008**: System MUST display available spec instances in a selector component
- **FR-009**: System MUST allow users to switch between spec instances, updating all views accordingly
- **FR-010**: System MUST remember the last opened project and spec instance between sessions

**Artifact Display**
- **FR-011**: System MUST render markdown files with proper formatting (headers, lists, code blocks, tables)
- **FR-012**: System MUST display spec.md content when Specify step is selected
- **FR-013**: System MUST display plan.md, research.md, data-model.md, and contract files when Plan step is selected, using a tabbed interface with one tab per artifact file
- **FR-014**: System MUST display tasks.md when Tasks step is selected
- **FR-015**: System MUST display appropriate content for each extended workflow step (Test, Push, PR, Bug Fix), using tabbed interface when multiple artifacts exist
- **FR-035**: System MUST display a file tree of source files created/modified for the current spec when Implement step is selected
- **FR-036**: System MUST provide a read-only source code viewer (no diff) when a file is selected in the Implement step file tree
- **FR-037**: System MUST support syntax highlighting in the source code viewer based on file type

**Checklist Functionality**
- **FR-016**: System MUST parse markdown checkboxes (`- [ ]` and `- [x]`) from artifact files
- **FR-017**: System MUST render checkboxes as interactive elements that users can toggle
- **FR-018**: System MUST persist checkbox state changes back to the source markdown file
- **FR-019**: System MUST calculate and display completion percentage for checklist sections
- **FR-020**: System MUST surface task completion status from tasks.md prominently in Tasks step

**Terminal Integration**
- **FR-021**: System MUST provide ability to open new terminal sessions
- **FR-022**: System MUST allow multiple concurrent terminal sessions
- **FR-023**: System MUST support switching between open terminal sessions
- **FR-024**: System MUST execute shell commands and display output in the terminal
- **FR-025**: System MUST support terminal sessions running in background while user works on other areas
- **FR-032**: System MUST display terminal area as a bottom panel below the detail pane
- **FR-033**: System MUST allow users to resize the terminal panel height by dragging
- **FR-034**: System MUST allow users to collapse/expand the terminal panel

**Extended Workflow Steps**
- **FR-038**: System MUST display a placeholder or test results summary when Test step is selected, showing test pass/fail counts if available
- **FR-039**: System MUST display git status information when Push step is selected, including current branch, staged/unstaged changes, and ahead/behind counts

**GitHub Integration**
- **FR-026**: System MUST connect to GitHub to retrieve PR information for the current branch
- **FR-027**: System MUST display PR review comments with file context and author information
- **FR-028**: System MUST display PR status checks and CI results
- **FR-029**: System MUST retrieve and display GitHub issues for the repository
- **FR-030**: System MUST allow filtering/selection of issues for bug fix workflow (by state: open/closed, by labels)
- **FR-031**: System MUST detect network connectivity status and disable GitHub-dependent steps (PR, Bug Fix) when offline, displaying a clear "No connection" message

### Key Entities

- **Project**: Represents a repository or folder containing `.specify/` and `specs/` directories; serves as the container for all work
- **SpecInstance**: A single feature specification within a project, identified by its directory name (e.g., `001-speckitui-core`); contains all artifacts for one feature
- **Artifact**: A file (typically markdown) produced by a workflow step; includes spec.md, plan.md, tasks.md, etc.
- **WorkflowStep**: One stage in the SDD process; has associated artifact types and display requirements
- **ChecklistItem**: A single checkbox item parsed from markdown; has checked/unchecked state and belongs to a section
- **TerminalSession**: An active terminal process; has input/output streams and running state
- **PRFeedback**: A collection of review comments and status checks from a GitHub pull request
- **GitHubIssue**: An issue from the repository; has title, body, labels, and state

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Users can navigate between all 8 workflow steps in under 1 second per click
- **SC-002**: Users can open a project and view spec artifacts within 3 seconds of folder selection
- **SC-003**: Users can switch between spec instances and see updated content within 2 seconds
- **SC-004**: 95% of markdown content renders correctly with proper formatting on first display
- **SC-005**: Users can toggle a checkbox and see the change persisted within 1 second
- **SC-006**: Users can open a new terminal session within 2 seconds
- **SC-007**: GitHub PR data loads and displays within 5 seconds of selecting the PR step (with valid authentication)
- **SC-008**: Task completion percentage is accurate within 1% of actual checkbox state
- **SC-009**: Users report the UI layout is "intuitive" in 80% of usability feedback
- **SC-010**: Users can complete a full specify→plan→tasks→implement cycle without leaving the application

## Clarifications

### Session 2026-01-16

- Q: When a workflow step has multiple artifacts (e.g., Plan step has plan.md, research.md, data-model.md, and contracts/), how should the detail pane organize and display them? → A: Tabbed interface with one tab per artifact file
- Q: How should the application behave when there is no network connectivity (offline mode) for GitHub-dependent features (PR step, Bug Fix step)? → A: Disable GitHub steps entirely when offline; show "No connection" message
- Q: What is the intended deployment model for this application? → A: Tauri desktop app (runs locally, direct file system access)
- Q: How should the terminal area be positioned relative to the main content, and should it be resizable/collapsible? → A: Bottom panel, resizable and collapsible (like VS Code integrated terminal)
- Q: For the "Implement" workflow step, what content should be displayed in the detail pane? → A: File tree of source files created/modified for this spec, with a read-only source viewer (no diff)

## Assumptions

1. **Target Platform**: Tauri-based desktop application; runs locally with direct file system access; mobile is out of scope for initial version
2. **Authentication**: GitHub authentication will use standard OAuth flow or personal access tokens
3. **File System Access**: Application has read/write access to the local file system for project folders
4. **Terminal**: Terminal sessions use the system's default shell (bash, zsh, PowerShell, etc.)
5. **Single User**: Application is designed for single-user local development; collaboration features are out of scope
6. **Spec-Kit Structure**: Projects follow the standard spec-kit directory structure (`.specify/`, `specs/`)
7. **Markdown Standard**: Artifact files use GitHub-flavored markdown syntax
